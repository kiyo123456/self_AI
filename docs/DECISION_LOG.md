### [2026-03-15] ExperienceのID管理方法

**背景・課題**  
Experience エンティティの ID をどこで発番・管理するか。
RDF の URI として使うため UUID が適切だが、「誰が発番するか」が不明確だった。

**検討した選択肢**  
- A案：Oxigraph に全てを任せる（RDF の blank node を使う）  
  メリット：シンプル  
  デメリット：Aggregate の ID をドメインが持てなくなる。DDD の原則に反する。

- B案：MySQL で Experience テーブルを持ち、UUID を発番する  
  メリット：ドメインが ID を持つ。エンティティ台帳として管理できる。  
  デメリット：MySQL と Oxigraph の二重管理になる。

**判断と理由**  
B案（MySQL で発番）を選択。
理由：「Aggregate のライフサイクル管理はドメインの責務」という DDD の原則を優先した。
Oxigraph はあくまで関係性を保存する場所であり、エンティティの存在を記録する場所ではない。

**この判断から学んだこと**  
RDF を使うからといって、全てを RDF で管理する必要はない。
「エンティティの台帳（MySQL）」と「関係性のグラフ（RDF）」を明確に分けることで、DDD の原則を守りながら RDF の強みを活かせる。

### [2026-03-15] スキルを推論で導出する設計

**背景・課題**  
ユーザーが「スキル」を直接入力するか、経験から推論で導出するか。

**検討した選択肢**  
- A案：ユーザーがスキルを入力する  
  メリット：実装がシンプル  
  デメリット：スキルは主観的で、バイアスがかかる。自己分析ツールとして価値が低い。

- B案：経験から推論で導出する（InferredSkill）  
  メリット：LLM による客観的な分析が可能。「気づいていない強み」を発見できる。  
  デメリット：推論の信頼性をどう担保するかが課題になる。

**判断と理由**  
B案（推論で導出）を選択。
理由：このプロジェクトの本質的価値は「RDF + LLM による推論」にある。
スキルを手入力にすると、単なる経歴管理ツールになってしまう。

**この判断から学んだこと**  
「スキル」という言葉を安易に使うと、ユーザー入力のスキルと推論で導出されたスキルが混同される。
だから `InferredSkill` という明確な名前をつけた。ユビキタス言語の重要性を実感した。

### [2026-03-20] Experience の概念設計（A案 vs B案）

**背景・課題**  
「ガクチカ」に相当するユビキタス言語を決めるにあたり、
時期（大学 / 高校 / 幼少期）で概念を分けるか、一つの概念にまとめるかを判断する必要があった。

**検討した選択肢**  
- A案：時期で概念を分ける（UniversityActivity / LifeExperience）  
  メリット：大学時代の経験と人生全体の経験を明確に区別できる  
  デメリット：推論のインプットとして時期は本質ではない。概念が増えて複雑になる。

- B案：一つの概念にまとめる（Experience）  
  メリット：ドメインがシンプル。推論ロジックも一つで済む。  
  デメリット：時期で絞り込みたい場合に属性設計が別途必要になる。

**判断と理由**  
B案（Experience 一つ）を採用。
人事が見ているのは「いつの経験か」ではなく「その経験から何を考えたか・どう活かすか」。
「活動」と「日記」の境界も曖昧であり、無理に分けるより一つにまとめた方がドメインが正直。

**この判断から学んだこと**  
人によって、自分のポテンシャルをアピールする時期はグラデーション。時期で区切ることはできない。
相手が何をみているのか把握するとこっちも何をすればいいのか自ずと見えてくる

### [2026-03-20] Experience の時期属性をMVPで省く（YAGNI適用）

**背景・課題**  
Experience に `period` 属性（大学 / 高校などのカテゴリ）を持たせるか検討したが、
複数タグを許可すると「大学時代の経験かどうか」の判定が曖昧になる問題が発生した。

**検討した選択肢**  
- A案：日付（started_at / ended_at）で持つ  
  メリット：グラデーションをそのまま表現できる  
  デメリット：ユーザーが正確な日付を覚えていないことが多い

- B案：カテゴリラベルで持つ（複数タグ可）  
  メリット：シンプルで実装しやすい  
  デメリット：複数タグ時に「大学時代の経験か」の判定ロジックが曖昧になる

- C案：MVPでは省く（YAGNI）  
  メリット：ドメインをシンプルに保てる。実際に使って検証できる  
  デメリット：後から追加する際に設計変更が必要になる可能性がある

**判断と理由**  
C案（YAGNI）を採用。
机上の議論より実際に動かして検証する方が有効。
「なぜ省いたか」の理由が言えることが重要で、「考えていなかった」ではなく「意図的に省いた」判断。

### [2026-04-09] 「常にアップデートされ続ける設計」の根拠を言語化

**背景・課題**  
VISION.md に「常にアップデートされ続ける相談相手」という将来像を書いていたが、
「なぜ一度の分析では不十分なのか」の根拠が言語化できていなかった。

**思考プロセス**  
- 自分のcoreとなる特性は変わらない
- しかし経験を積むなかで化学反応が起き、解像度が上がり、別の何かに変わる可能性がある
- → 経験の蓄積とともに推論が深まり続ける設計が必要

**この判断がアーキテクチャに与える影響**  
「常にアップデート」という要件は、以下の設計判断を必然的に導く。

| 要件 | 設計上の対応 |
|------|------|
| coreは変わらない | Experienceとして蓄積し続ける（削除不可の制約） |
| 化学反応・解像度が上がる | InferenceSessionを繰り返し実行することでInferredSkill / InferredValueが更新される |
| 別の何かに変わる | Strengthがより抽象度高く再導出される |

つまり「経験を蓄積するほど推論精度が上がる」RDFの設計は、
「一度わかれば終わりではなく、常に更新が必要」というプロダクト理念から**必然的に導かれている**。

**この判断から学んだこと**  
技術選択（RDF + 非同期推論）の根拠が、プロダクト理念まで遡って説明できるようになった。
面接で「なぜRDFを使ったか」を問われたとき、技術的な理由だけでなく、
「ユーザーの自己理解が経験とともに深まるというプロダクトの本質から来ている」と答えられる。
