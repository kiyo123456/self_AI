### [2026-03-15] ExperienceのID管理方法

**背景・課題**  
Experience エンティティの ID をどこで発番・管理するか。
RDF の URI として使うため UUID が適切だが、「誰が発番するか」が不明確だった。

**検討した選択肢**  
- A案：Oxigraph に全てを任せる（RDF の blank node を使う）  
  メリット：シンプル  
  デメリット：Aggregate の ID をドメインが持てなくなる。DDD の原則に反する。

- B案：MySQL で Experience テーブルを持ち、UUID を発番する  
  メリット：ドメインが ID を持つ。エンティティ台帳として管理できる。  
  デメリット：MySQL と Oxigraph の二重管理になる。

**判断と理由**  
B案（MySQL で発番）を選択。
理由：「Aggregate のライフサイクル管理はドメインの責務」という DDD の原則を優先した。
Oxigraph はあくまで関係性を保存する場所であり、エンティティの存在を記録する場所ではない。

**この判断から学んだこと**  
RDF を使うからといって、全てを RDF で管理する必要はない。
「エンティティの台帳（MySQL）」と「関係性のグラフ（RDF）」を明確に分けることで、DDD の原則を守りながら RDF の強みを活かせる。

### [2026-03-15] スキルを推論で導出する設計

**背景・課題**  
ユーザーが「スキル」を直接入力するか、経験から推論で導出するか。

**検討した選択肢**  
- A案：ユーザーがスキルを入力する  
  メリット：実装がシンプル  
  デメリット：スキルは主観的で、バイアスがかかる。自己分析ツールとして価値が低い。

- B案：経験から推論で導出する（InferredSkill）  
  メリット：LLM による客観的な分析が可能。「気づいていない強み」を発見できる。  
  デメリット：推論の信頼性をどう担保するかが課題になる。

**判断と理由**  
B案（推論で導出）を選択。
理由：このプロジェクトの本質的価値は「RDF + LLM による推論」にある。
スキルを手入力にすると、単なる経歴管理ツールになってしまう。

**この判断から学んだこと**  
「スキル」という言葉を安易に使うと、ユーザー入力のスキルと推論で導出されたスキルが混同される。
だから `InferredSkill` という明確な名前をつけた。ユビキタス言語の重要性を実感した。
