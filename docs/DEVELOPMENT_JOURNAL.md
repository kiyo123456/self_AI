## 記録の方針

- 正解を書く場所ではない。試行錯誤のプロセスを書く場所。
- 「最初はAだと思ったけど、Bだった」という差分を記録する。
- AIに指摘された差分を内省し、次のタスクの精度を上げる材料にする。

### [2026-03-15] ユビキタス言語をゼロから作るプロセス体験

**今日やったこと**  
AIとの壁打ちを通じて、ユビキタス言語を「答えを写す」のではなく「自分で考えて導き出す」プロセスを体験した。

**やり取りの流れ（思考プロセスの記録）**

1. **ドメインの専門家を特定した**  
   → 就活生本人と人事担当者の2者。両者が共通で使える言葉を探すことがユビキタス言語の出発点。

2. **原石となる言葉を列挙した**  
   → 自己分析 / 興味ある分野 / 趣味 / 好きなこと / スキル / 学生時代がんばったこと / 大学 / ガクチカ / キャリア像

3. **「スキル」と「ガクチカ」を「事実 vs 解釈」「入力 vs 推論」の軸で分類した**  
   → ガクチカは具体レベルでは事実・抽象レベルでは解釈。スキルは推論の産物。

4. **「スキルをユーザー入力可能にすると何が壊れるか」を考えた**  
   → ユーザー入力スキルとシステム推論スキルが混在し、どちらを信頼するか曖昧になる問題が起きる。
   → だからスキルはシステムのみが生成するという制約が必要。

5. **「名前が制約を伝えるべき」という原則を適用した**  
   → 「スキル」では制約が伝わらない → 「InferredSkill（推論スキル）」に昇格。

6. **「ガクチカ」の用語問題に気づいた**  
   → 「ガクチカ」は就活生の間でしか使われない。人事は「学生時代頑張ったこと」と言う。
   → ユビキタス言語は専門家間で共通でなければならないため、ガクチカはNG。

7. **「学生時代」の範囲問題を発見した（→ 次回の宿題）**  
   → 就活で話すのは主に大学時代だが、価値観・強みを掘り下げると幼少期まで遡る。
   → A案：時期で概念を分ける（UniversityActivity / LifeExperience）
   → B案：時期を「属性」として持つ一つの概念にまとめる（Experience）
   → **この問いに対して、まだ答えを出していない。次回ここから再開。**

**今日確定したユビキタス言語（自分で導き出したもの）**

```
用語：InferredSkill（推論スキル）

定義：ユーザーの具体的な活動や取り組み・成果物から
      システムによって推論・抽出される抽象的な概念

制約：ユーザー本人は入力・変更できない
```

**学んだこと**  
- ユビキタス言語は用語・定義・制約で表現する
- 制約の視点がなかった。制約も考えることで、よりこの用語の位置付けが明確になるからいいなと感じた。
- 案がいくつかある場合、それぞれどのような問題点があるかを考えると取捨選択しやすい
- 「ガクチカ」のように、片方の専門家しか使わない言葉はユビキタス言語になれない

**次にやること（次回の再開ポイント）**  
A案・B案のトレードオフを考える。
特に「将来、システムが推論するとき」の視点でどちらが適切かを判断して、
「ガクチカ」に相当するユビキタス言語の用語名と定義を完成させる。

### [2026-03-20] Experience のユビキタス言語を完成させる

**今日やったこと**  
前回の宿題だった「ガクチカに相当するユビキタス言語」を完成させた。
A案（時期で概念を分割）とB案（一つの概念にまとめる）のトレードオフを検討し、
最終的に Experience のユビキタス言語定義を自分の言葉で書き上げた。

**思考プロセスの記録**

1. システムが推論するシーンを言語化した  
   → 「活動から InferredSkill を推論」「複数活動から価値観を抽出」「日記から価値観を創出」の3パターン

2. 「活動（事実）」と「日記（主観）」を別概念にすべきか検討した  
   → 活動の記録にも感じたこと・学んだことが混在する。境界が曖昧なので、一つの概念にまとめた。

3. A案（UniversityActivity / LifeExperience）の「時期で分割」を検討した  
   → 推論に時期は直接関係しない。「経験の中身」で推論できれば十分。→ B案（Experience 一つ）を採用。

4. 時期の属性（period）をどう持つか検討した  
   → カテゴリラベル案・複数タグ案を検討したが、複数タグにすると「大学時代の経験」の判定が曖昧になる。
   → YAGNI で一旦省く。実際に使って必要を感じたら追加する。

5. プロダクトビジョンを言語化した  
   → 「ユーザーが日常の記録をとりあえず入力し続けることで、RDF を使って AI がスキル・価値観・強みを推論する。自分が気づかなかった一面を発見し、就活の自己分析に活用できる。」

6. Experience のユビキタス言語定義を完成させた

**今日確定したユビキタス言語**

```
用語：Experience（経験）

定義：日々の活動・日記・その中で感じたこと・気づいたことが蓄積されたデータのこと。
      量を蓄積することで価値が生まれる。

制約：ユーザー本人のみ入力できる
      削除できる
      蓄積された経験の内容を変更できない
      記録日時は入力時に自動で付与され、変更できない
```

**学んだこと**  
制約には、そのドメインでエンティティができることできないことを決めるためにある。

**次にやること**
- DESIGN.md の Experience の定義を今日確定した内容に更新する
- 次のユビキタス言語（Value / Triple など）の定義に進む
- Experience エンティティの実装に入る

### [2026-04-08] InferredValue・Triple のユビキタス言語を確定させる

**今日やったこと**
Value と Triple のユビキタス言語定義を壁打ちで完成させた。
あわせて DESIGN.md の Experience 定義を前回確定した内容に更新した。

**思考プロセスの記録**

1. **Value は「事実」か「推論」かを考えた**
   → 推論に近いと感じた。経験から導き出されるものだから。

2. **「両方扱いたい」という発想が生まれた**
   → ユーザーが自己申告した価値観 ＋ システムが推論した価値観の両方を扱いたい。

3. **スキルの問題と同じ状況に気づいた**
   → ユーザー入力とシステム推論が混在すると、どちらを信頼するか曖昧になる（前回と同じ問題）。

4. **ユーザー申告の価値観は Experience に含められると判断した**
   → 「チームワークを大切にしていると気づいた」は日々の気づきであり、Experience の定義に自然に収まる。
   → Experience の定義がさらに広がった：活動・日記・感じたこと・気づいたこと・自分の信念の自己申告

5. **システムが出力する価値観を InferredValue と命名した**
   → InferredSkill に倣い、制約を名前で伝える。

6. **InferredValue と InferredSkill の差分を言語化した**
   → InferredSkill：活動・取り組み・成果物（客観的事実）から推論
   → InferredValue：活動・取り組み・成果物 ＋ 主観的な考察から推論
   → 価値観はスキルと違い、ユーザーの内面的な考察も推論材料にするという意図的な設計判断。

7. **Triple の生成者とタイミングを考えた**
   → 誰が：システムが自動生成する
   → いつ：Experience が登録されたタイミング
   → 1つの Experience から複数のトリプルが生まれることも確認した（例：所属した組織 × 複数）

**今日確定したユビキタス言語**

```
用語：InferredValue（推論価値観）
定義：ユーザーの具体的な活動や取り組み・成果物・主観的な考察をもとに
      システムによって推論・抽出される抽象的な概念
制約：ユーザー本人は入力・変更できない
```

```
用語：Triple（トリプル）
定義：<主語, 述語, 目的語> の形式で表現され、
      システムが作成する知識の最小単位
制約：ユーザーは作成・変更できない
      Experience が登録されたタイミングでシステムが自動生成する
```

**学んだこと**
- 前回学んだ「ユーザー入力 vs システム推論の混在問題」を、今回は自分から気づいて適用できた
- 概念を分離するとき、「どちらの入力を信頼するか」という問いが判断軸になる
- 定義の差分（InferredSkill vs InferredValue）を言語化することで、なぜ2つの概念が必要かを説明できる

**次にやること**
- InferenceSession / AnalysisQuery のユビキタス言語定義に進む
- または Phase 1（Laravel プロジェクトセットアップ）に入る

### [2026-04-12] Strength / InferenceSession / AnalysisQuery のユビキタス言語を確定させる

**今日やったこと**
残っていた3概念のユビキタス言語定義を壁打ちで完成させた。

**思考プロセスの記録**

#### Strength
1. InferredSkill との違いを考えた  
   → InferredSkill は表面から見えるもの（トレーニングで強化できる）  
   → Strength はスキルを支える根底。思考・行動の無意識的な部分。自分では当たり前だと思っている部分。
2. 生成経路を考えた  
   → 最初「複数の経験から」と言ったが、「複数のスキルから」に補正した  
   → 理由：「要件定義したことがある」だけでは「構造化したい衝動」は見えない。複数の InferredSkill のパターンで初めて強みが浮かび上がる。
3. ユーザーが変更できるかを考えた  
   → 客観的に出された強みをユーザーが変更できてしまうのはプロダクトの意味がない → 変更不可。

#### InferenceSession
1. エンティティかバリューオブジェクトかを考えた  
   → 状態が変化する（情報スキャン中 → 分析中 → 完了/失敗）→ ライフサイクルがある → エンティティ。
2. 1ユーザーが複数持てるかを考えた  
   → 推論結果は上書きされず蓄積される → 複数持てる。これが「常にアップデートされ続ける」ビジョンと一致する。

#### AnalysisQuery
1. 誰が作るかを考えた → ユーザー。
2. 毎回 InferenceSession を走らせる必要があるか考えた  
   → InferenceSession は非同期で重い処理。AnalysisQuery は既存の推論結果を材料にして答えれば十分。  
   → 「過去に走らせた InferenceSession をもとに回答する」という設計判断。
3. MVP の AnalysisQuery は「企業サイトを投げて相性を聞く」は含まない  
   → それは将来の「自分を知っている相談相手」の世界。

**今日確定したユビキタス言語**

```
用語：Strength（強み）
定義：複数の経験から AI によって推論された複数の InferredSkill を材料に、さらに AI が推論した結果として導き出されるもの。
      思考や行動の無意識的な部分から始まる、自分では当たり前だと思っている部分であり、スキルの土台となる概念。
制約：ユーザーは入力・変更できない
```

```
用語：InferenceSession（推論セッション）
定義：ユーザーが「推論を実行」し、InferredSkill / Strength が生成されるまでの一連の操作
制約：1ユーザーが複数持てる／推論結果は上書きされず新しい Session として蓄積される
状態：情報スキャン中 → 分析中 → 完了 / 失敗
```

```
用語：AnalysisQuery（分析クエリ）
定義：推論結果に対して「私の強みは？」などの問いを投げる操作。過去に走らせた InferenceSession の結果をもとに回答する。
制約：ユーザーが作成する／新たな InferenceSession は起動しない
```

**学んだこと**
- Strength を「InferredSkill の根底にあるもの」と言語化できた。具体例（「要件定義できる」←「構造化したい衝動」）があると説明しやすい。
- AnalysisQuery と InferenceSession の境界を意識できた。問いを投げるたびに推論を走らせると UX が悪い → 既存結果を再利用する設計の理由を自分の言葉で説明できる。
- MVP の範囲を「将来像」と切り分けながら考えられた。

**次にやること**
- ユビキタス言語が全7概念揃ったので、Phase 1（Laravel プロジェクトセットアップ）に入る
